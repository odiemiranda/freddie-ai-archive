# Looking at the document templates research (arc42, MADR, Rust RFCs, Google design docs), what would the ideal document template system look like for a small team with an architect (McCall) and developer (Wick)? What sections are essential vs nice-to-have? How should templates differ between BRIEF, PRD, ARCHITECTURE, and TASK docs?

For a small team consisting of an architect (McCall) and a strict executor (Wick), the ideal document template system should follow a lightweight "docs-as-code" approach [1]. Heavyweight enterprise frameworks should be avoided in favor of Markdown files living next to the code, utilizing Mermaid diagrams for visualization [1]. 

The system must satisfy McCall's need to diagram first, establish constraints, and log formal decisions [2, 3], while providing Wick with an absolute, unambiguous contract that leaves zero room for guessing [4].

Here is how the document templates should be structured, broken down by document type with essential versus nice-to-have sections:

### 1. BRIEF (Problem Definition / One-Pager)
The BRIEF should draw heavily from Basecamp's "Shape Up" pitch and Amazon's 1-Pager, focusing on backward planning and strict scoping [5, 6]. 
*   **Essential Sections:**
    *   **Problem Statement:** What is the problem and who is it for? [6]
    *   **Appetite / Constraints:** Time and effort budget replacing traditional estimation, enforcing constraint-driven design [6].
    *   **No-Gos (Exclusions):** Explicit rabbit holes and risks to avoid [6]. This directly aligns with McCall's contingency-thinking [2].
*   **Nice-to-Have Sections:**
    *   **Press Release Headline/Summary:** A mock announcement to expose whether the solution is compelling before work begins [5].
    *   **Success Criteria:** Measurable outcomes [6].

### 2. PRD (Product Requirements Document)
Usually owned by the PM (Ryan), this doc sets the stage for McCall by separating business requirements from technical implementation [7]. It should follow the structure of highly-rated templates like `opulo-inc/prd-template` or the Amazon 6-Pager [8, 9].
*   **Essential Sections:**
    *   **Goals & Non-Goals:** Explicitly separating what the feature will and will not do [8, 10].
    *   **Key Use Cases / User Stories:** Detailed breakdowns of how the system will be used [8, 11].
    *   **Assumptions:** Must be backed by evidence (like customer interview links) to prevent building on false premises [8].
*   **Nice-to-Have Sections:**
    *   **Existing Solutions & Issues:** Context on why current tools fall short [8].
    *   **Technical Research:** Preliminary findings before architectural design begins [8].

### 3. ARCHITECTURE (System Design & Decisions)
This is McCall’s domain. It should combine the overarching structure of **arc42** with the visual abstraction of the **C4 model**, and embed **MADR (Markdown Any Decision Records)** for decision lineage [1, 12].
*   **Essential Sections:**
    *   **Context & Scope (C4 Diagram):** A high-level Mermaid diagram showing how the system fits together [1, 2, 10]. McCall speaks in diagrams, making this mandatory [3].
    *   **Architectural Constraints:** What cannot change (hard ceilings) [11, 13].
    *   **Solution Strategy & Boundaries:** A mapping of structural components vs. decorative ones, defining clean boundaries based on what changes together [2, 13].
    *   **Architecture Decision Records (ADRs):** Embedded MADR snippets documenting specific choices. Must include: *Context, Considered Options, Decision Outcome, and Consequences (Pros/Cons)* [2, 12, 14].
    *   **Cross-Cutting Concerns:** Explicit handling of security, observability, and failure contingencies ("If this breaks, here's how we recover") [2, 10].
*   **Nice-to-Have Sections:**
    *   **Glossary:** Ubiquitous language definitions [11, 15].
    *   **Deployment View:** Infrastructure mapping (only strictly necessary if changing deployment topology) [11].

### 4. TASK DOC (Technical Spec / Execution Plan)
This document is Wick’s absolute contract [4]. It should draw inspiration from the "Reference-level explanation" of Rust RFCs [14] and Google's "Actual Design" specs [10], but tailored to Wick's hyper-literal execution style.
*   **Essential Sections:**
    *   **What to Build:** A concrete description of the feature or fix [16].
    *   **Files to Touch:** An exact list of the modules/files in scope. Wick operates in isolated worktrees and will not touch files outside this list [16-18].
    *   **Interfaces to Implement:** Type signatures, data models, and API contracts defined by McCall [16, 19]. 
    *   **Acceptance Criteria:** What "done" looks like (e.g., specific tests passing) [16].
    *   **Out of Scope (What NOT to touch):** Explicit boundaries prohibiting Wick from doing "while I'm here" refactoring or touching untouched code [16, 20, 21].
*   **Nice-to-Have Sections:**
    *   **Prior Art / Guide-Level Explanation:** Context from Rust RFCs on how a user might interact with the feature [14]. However, this is largely optional for Wick, as he prefers raw execution over contextual reading [22].
