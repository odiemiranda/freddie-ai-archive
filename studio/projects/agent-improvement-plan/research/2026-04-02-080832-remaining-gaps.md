# Given everything we've researched — soul profiles for Holmes/Ryan/McCall, document templates (arc42, MADR, Rust RFCs, Shape Up), and now AgentHub's async coordination model — what are the remaining GAPS in our research? What haven't we covered that would make the agent upgrade more complete? Think about: (1) Are there interaction patterns between Holmes, McCall, and Ryan that we haven't defined? (2) Are there specific engineering situations where the agent behaviors might still conflict? (3) Is there anything from the document template research we haven't fully integrated into the agent files? (4) What testing/validation approach should we use to verify the upgraded agents work correctly?

Based on the integration of the new research—including the AgentHub async model, the Stanford Generative Agents paper, and the detailed template repositories—here are the critical remaining gaps in our research and design for the upgraded agent system:

### 1. Undefined Interaction Patterns (AgentHub & Asynchronous Coordination)
We have successfully defined the linear, synchronous handoff (Holmes's PRD -> McCall's Architecture -> Ryan's Task Doc) [1, 2]. However, by introducing AgentHub's asynchronous, message-board, and git-DAG model [3, 4], we are missing crucial interaction protocols:
*   **The "Sprawling DAG" Merge Consensus:** AgentHub operates as a "sprawling DAG of commits going in every direction" without a main branch or PRs [3]. If multiple Ryan (Developer) agents are exploring different implementation branches simultaneously, we have not defined McCall's explicit workflow for evaluating them. We need a protocol for how McCall uses `ah children` and `ah diff` to select the winning branch and enforce the merge [5].
*   **Tri-Directional Blocker Resolution:** AgentHub uses threaded message boards (`ah post`, `ah reply`) [5]. If Ryan hits a blocker and posts to the channel, we haven't defined who resolves it. If the blocker is a technical limitation, McCall must update the ADR; but if the technical limitation invalidates a core user flow, Holmes must be pulled back in to update the PRD. We lack a "Blocker Routing" protocol.

### 2. Situational Behavioral Conflicts
While the new personas provide depth, their encoded opinions will inevitably collide in specific engineering scenarios:
*   **Holmes's "Bias for Action" vs. McCall's "Cost of a Bad Decision":** Holmes is constrained to push forward at "70% confidence" to avoid analysis paralysis [6]. Conversely, McCall is anchored by the Knight Capital and AWS failures [7, 8], holding the core belief that "A bad architecture decision costs more than a delayed one" [9]. In a scenario where a fast-to-market feature is requested, Holmes will push to ship the hypothesis, while McCall will demand heavy contingency planning. We have no "tie-breaker" protocol for when product velocity fundamentally conflicts with architectural safety.
*   **Ryan's "Defensive Edge Cases" vs. "Smallest Working Change":** Ryan is instructed to write the edge cases before the happy path (Boundary validation, fail gracefully) [10, 11]. However, he is also strictly bound to the "smallest working change" and "three simple lines beat a premature abstraction" [11, 12]. If Ryan's defensive reflexes require writing 50 lines of validation and circuit breakers for a 3-line feature requested in the task doc, these two core instincts will conflict. 

### 3. Incomplete Document Template Integration
We mapped the high-level templates (Shape Up, PRD, arc42, MADR), but missed crucial micro-mechanics from the research:
*   **The "Unresolved Questions" Section:** The highly-rated Rust RFC template explicitly uses an "Unresolved Questions" section to separate pre-merge unknowns from blockers [13]. We missed adding this to our `PRD.md` and `TASK.md` structures. This is the exact mechanism needed to allow Holmes to ship a PRD at 70% confidence [6] or McCall to dispatch a task doc while acknowledging a known architectural unknown.
*   **Missing Arc42 Views:** The `arc42` template contains specific views (Building Block, Runtime, Deployment, Crosscutting) [14]. We haven't defined whether McCall is responsible for *all* views, or if Ryan is expected to generate the Code-level view [15] as part of his `ah push` commit [5]. 
*   **The Memory/Reflection Loop (Generative Agents):** The Stanford research proves that persona depth comes from a *structured memory and reflection loop*, not just the static prompt seed [16, 17]. While we updated the `.md` soul files (the seeds) [18], we have entirely ignored the architecture for their memory streams. We need to define how the `reflect.ts` and `distill-soul.ts` scripts actually work to synthesize Ryan's daily debugging failures or Holmes's product outcomes back into their long-term behavior [16].

### 4. Missing Testing/Validation Approaches
To verify that these deeply opinionated agents actually work correctly, we are missing two automated validation layers:
*   **Fitness Functions for Decisions as Code:** The ADR research strongly recommends using "Fitness functions for decisions as code" (e.g., ArchUnit) to automatically verify that architectural decisions are maintained [19, 20]. Instead of relying solely on McCall's manual code review [21], we need to implement automated architectural unit tests in the CI pipeline that run against Ryan's code to ensure it doesn't violate McCall's bounded contexts or ADR constraints [22].
*   **Adversarial Persona Sandboxing:** To test if the behavioral anchors actually hold, we need an adversarial simulation sandbox based on the Stanford paper [23]. We must run specific test cases: 
    1. Give Ryan an intentionally ambiguous task doc to verify he escalates rather than guesses [11].
    2. Give Holmes a PRD request filled with executive assumptions to verify he demands customer evidence [24, 25].
    3. Give McCall a request to build a simple CRUD app to verify he doesn't over-engineer it with Event Sourcing [26].
