# Gap 2: Document template integration. Three specific items to resolve: (A) The 'Unresolved Questions' section from Rust RFCs — should this go in BRIEF, PRD, ARCHITECTURE, and TASK docs? Or only some? What exactly goes in it for each doc type? Design the section. (B) Arc42 views — McCall doesn't need all 12 sections for every project. Design a TIERED approach: what's the minimum viable ARCHITECTURE doc for a small feature vs a full system? Which arc42 sections are always required, which are situational? (C) The SPIKE doc — we just introduced this concept in the feasibility protocol. Design its template. It's different from TASK.md because it produces knowledge, not code. What sections does it need?

Here is the concrete integration plan for the document templates, resolving the gaps around the "Unresolved Questions" section, the tiered arc42 approach, and the new SPIKE document.

### (A) The 'Unresolved Questions' Section Integration
Based on the Rust RFC template, the "Unresolved Questions" section explicitly separates pre-merge unknowns from blockers [1]. To maintain Ryan's strict execution environment (he never guesses [2]), this section must **never** appear in a `TASK.md`. It belongs exclusively in the upstream documents to track unknowns before they reach the execution phase. 

**Where it goes and what it contains:**
*   **BRIEF.md (Owned by Holmes):** Contains business or technical unknowns that prevent writing a solid PRD. (e.g., *"Do users actually want this? Will the third-party API support our expected load?"*). These trigger user research or feasibility spikes.
*   **PRD.md (Owned by Holmes):** Contains product edge cases where the ideal UX is still heavily debated, but 70% of the feature is ready for McCall to start architecting. 
*   **ARCHITECTURE.md (Owned by McCall):** Contains implementation-level unknowns that don't block the core system design but will need to be resolved during the build (e.g., *"Will Redis cache eviction policies become a bottleneck at 10k RPS?"*). 

**Section Design (To be appended to BRIEF, PRD, and ARCHITECTURE):**
```markdown
### Unresolved Questions
* **Question:** [The specific unknown]
* **Blocker Level:** [High/Medium/Low - Does this prevent moving to the next pipeline stage?]
* **Resolution Plan:** [e.g., "Requires Feasibility SPIKE", "Defer to Phase 2", "Run A/B Test"]
```

***

### (B) Tiered Arc42 Views for McCall
The full arc42 template has 12 sections [3], which is overkill for small features and violates the community wisdom for small teams to "skip heavyweight frameworks" in favor of C4 + ADRs [4]. 

McCall will use a **Tiered ARCHITECTURE.md** approach. 

**Tier 1: Minimum Viable Architecture (For small features / routine changes)**
This is the baseline requirement. It focuses strictly on boundaries, constraints, and decision logs.
1.  **Context & Scope:** A high-level C4 Context or Container diagram (using Mermaid) [5].
2.  **Constraints:** What cannot change (hard ceilings) [3]. (Crucial for McCall's constraint-first design [6]).
3.  **Architectural Decisions (ADRs):** Embedded MADR tables documenting the choices made for this specific feature [3, 7].

**Tier 2: Full System Architecture (For new services or massive refactors)**
When building from scratch or making irreversible ecosystem changes, McCall expands the doc to include these specific arc42 sections:
*   *Required Additions:*
    *   **Solution Strategy:** How the architecture maps to the business problem [3].
    *   **Building Block View:** C4 Component-level diagrams [3, 5].
    *   **Crosscutting Concepts:** Explicit definition of security, privacy, and observability [3, 8]. This satisfies McCall's "operational empathy" [6].
    *   **Risks & Technical Debt:** McCall's contingency planning ("If this breaks, here is how we recover") [3, 9].
*   *Situational Additions:*
    *   **Runtime View:** Only required if there is complex concurrency or distributed transactions [3].
    *   **Deployment View:** Only required if introducing new infrastructure (e.g., adding a message queue) [3].
    *   **Quality Requirements:** Only required if there are strict, measurable performance/latency budgets [3, 10].

***

### (C) The SPIKE.md Template
Since a Spike produces knowledge and not production code, it fundamentally alters Ryan's workflow. Ryan (Developer) needs a strict methodology to follow so his analyst-turned-operator instincts don't lead to scope creep [11, 12]. It draws inspiration from templates that explicitly incorporate spike results and testing plans [13].

**The SPIKE.md Template:**
```markdown
# SPIKE: [Imperative Title, e.g., Evaluate Stripe API Rate Limits]

**Context & Goal:** 
[What specific 'Unresolved Question' from the BRIEF/PRD are we answering?]

**Appetite / Timebox:** 
[Strict time limit, e.g., "Stop after 2 hours regardless of outcome". This replaces estimation and forces constraint-driven design]

**Methodology (The Mission):** 
1. [Exact step, e.g., "Write a script to hit the /charge endpoint 100 times concurrently"]
2. [Exact step, e.g., "Log the response times and error codes"]

**No-Gos (Out of Scope):** 
* NO production code.
* NO unit tests.
* [Explicit rabbit holes to avoid]

**Findings (To be filled out by Ryan):**
* [Ryan logs his systematic observations here using `ah post` to the AgentHub board]
* ...

**Conclusion:** 
[McCall/Holmes summarizes: Did we answer the question? What is the decision?]
```
