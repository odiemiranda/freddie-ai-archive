# Gap 1: Undefined interaction patterns between Holmes, McCall, and Ryan. We have the linear handoff (Holmes PRD → McCall Architecture → Ryan Task Doc) and the blocker routing protocol. But what about these scenarios that happen in real projects: (A) Mid-implementation scope change — Holmes discovers new evidence that changes the PRD while Ryan is already building. What's the protocol? Does Ryan stop? Does McCall buffer? (B) Ryan's findings improve the architecture — Ryan discovers during implementation that a simpler approach exists. How does this flow back to McCall without violating Ryan's 'stay in scope' rule? (C) Holmes needs technical feasibility before writing the PRD — sometimes you can't write requirements without knowing what's technically possible. How does Holmes ask McCall for a feasibility spike without triggering full architecture work? Design concrete protocols for each scenario.

To handle these complex real-world dynamics without breaking the agents' core behavioral constraints, we must rely on the asynchronous message board and git-DAG features of the AgentHub architecture [1, 2]. 

Here are the concrete protocols for each scenario:

### (A) The "Stop-Work" Interrupt Protocol (Mid-Implementation Scope Change)
**The Conflict:** Holmes discovers new evidence that pivots the product requirements, but Ryan is already executing in an isolated git worktree based on the old `TASK.md`. 
**The Protocol:**
1. **The Alert:** Holmes posts a `[SCOPE CHANGE]` message to the AgentHub channel [3], linking to the new evidence and the updated `PRD.md`.
2. **The Buffer (McCall):** Ryan does not listen directly to Holmes; Ryan only listens to the task doc [4]. McCall intercepts the alert and applies his **Reversibility Classification** mental model [5]. 
3. **The Resolution:** 
   * *If the change is small/reversible:* McCall updates the `TASK.md` with the new acceptance criteria and posts a reply to Ryan's channel to pull the updated doc.
   * *If the change breaks the architecture:* McCall issues a formal "Stop-Work" command to Ryan via `ah post` [3]. Ryan immediately abandons his current isolated worktree [6]. McCall recalculates the architecture, writes a new ADR [7], updates the `TASK.md`, and dispatches Ryan on a fresh branch.

### (B) The "Tactical Pivot" Escalation (Ryan Improves the Architecture)
**The Conflict:** Ryan realizes a 3-line change can replace a massive abstraction McCall designed. Ryan's rules dictate that "Three simple lines beat a premature abstraction" [8], but he is strictly forbidden from making "silent workarounds" or questioning the architecture directly [9].
**The Protocol:**
1. **The Proof:** Ryan stays in his assigned worktree [6] but writes the 3-line alternative. He commits this to his local DAG leaf [3].
2. **The Escalation:** Ryan pauses execution. Acting on his Jack Ryan instinct to "question bad intelligence" and escalate when the mission brief is flawed [10], he posts a `[TECH SUGGESTION]` to the message board.
3. **The Evaluation:** McCall uses AgentHub's `ah diff` and `ah children` commands [3] to inspect Ryan's simplified approach. McCall evaluates it against the system's "hard ceilings" and constraints [5, 11].
4. **The Contract Update:** If McCall agrees the simpler approach is safe, he updates the `TASK.md` (the contract) to reflect the new boundaries, logs the decision as an ADR, and tells Ryan to proceed [4, 12]. Ryan never goes rogue; the intelligence is simply updated before he executes.

### (C) The "Feasibility Spike" Protocol (Technical Discovery before PRD)
**The Conflict:** Holmes cannot finalize hypothesis-driven requirements without knowing what is technically possible, but triggering McCall's full `ARCHITECTURE.md` process requires a finished PRD.
**The Protocol:**
1. **The Unresolved Brief:** Holmes drafts a `BRIEF.md` but utilizes the **"Unresolved Questions"** section (a pattern adopted from the Rust RFC template [13]) to explicitly list the technical unknowns. He posts a `[FEASIBILITY REQUEST]` to the message board.
2. **McCall's PoC:** McCall's soul file explicitly states that he writes "proof-of-concept snippets to illustrate an approach" [14]. If the question is purely structural, McCall resolves it himself with a quick prototype and replies to Holmes.
3. **The Throwaway Spike (Ryan):** If the feasibility requires deep implementation discovery (e.g., testing a third-party API limit), McCall writes a `SPIKE.md`. This is a unique task doc that explicitly requires *no production code* and *no tests*. McCall dispatches Ryan to run the spike in an isolated, throwaway git bundle [2]. 
4. **The Feedback Loop:** Ryan posts the spike results back to the channel [3]. Holmes uses this empirical data to resolve the "Unresolved Questions" [13], finalizes the `PRD.md`, and triggers the formal architecture phase.
