# AAR-0006: Morning Briefing Integration

## Objective

Add a "Soul Updates" section to the morning briefing that shows:
1. Auto-applied soul changes since the last briefing
2. Staged proposals awaiting human review

This is the notification layer that closes the autonomous pipeline loop. The user sees what happened overnight without needing to check files manually.

## Context

**Current morning.ts behavior:** Displays greeting, tasks (from daily note), inbox counts, and suggestions. No awareness of soul changes or staged proposals.

**Target behavior:** After the tasks section and before suggestions, display a "Soul Updates" section when there are recent changes or pending proposals.

#### New function: `loadSoulUpdates()`

```typescript
interface SoulUpdateDisplay {
  applied: string[];     // formatted lines for auto-applied changes
  pending: string[];     // formatted lines for staged proposals needing review
  hasUpdates: boolean;
}

function loadSoulUpdates(): SoulUpdateDisplay {
  const applied: string[] = [];
  const pending: string[] = [];

  try {
    initBrain();

    // Get auto-applied changes from the last 24 hours
    // Use yesterday's date as the "since" filter
    const yesterday = new Date();
    yesterday.setDate(yesterday.getDate() - 1);
    const sinceDate = yesterday.toISOString().slice(0, 10); // YYYY-MM-DD

    const recentChanges = getRecentSoulChanges(undefined, { since: sinceDate, limit: 20 });

    for (const change of recentChanges) {
      if (change.status === "auto-applied") {
        applied.push(
          `+ ${capitalize(change.agent)}: ${change.changeSummary}`
        );
      }
    }
  } catch {
    // brain.db failure — skip soul change display
  }

  try {
    // Get pending staged proposals
    const proposals = getPendingProposals();

    for (const p of proposals) {
      const reason = p.proposal.reason.length > 60
        ? p.proposal.reason.slice(0, 57) + "..."
        : p.proposal.reason;
      pending.push(
        `? ${capitalize(p.agent)}: "${p.proposal.proposedChange.split('\n')[0].slice(0, 40)}" — ${reason}`
      );
    }
  } catch {
    // staged file read failure — skip pending display
  }

  return {
    applied,
    pending,
    hasUpdates: applied.length > 0 || pending.length > 0,
  };
}
```

## Acceptance Criteria

1. **Soul Updates section appears.** When there are auto-applied changes or pending proposals, the section displays between tasks and suggestions.
2. **Section does not appear when empty.** If no recent changes and no pending proposals, the section is omitted entirely.
3. **Applied changes show correctly.** Format: `+ {Agent}: {summary}`
4. **Pending proposals show correctly.** Format: `? {Agent}: "{change preview}" — {reason}`
5. **24-hour window.** Auto-applied changes are shown from the last 24 hours (not all time).
6. **Pending proposals are not time-windowed.** All pending proposals show regardless of age.
7. **Graceful failure.** If brain.db is unavailable or staged files can't be read, the section is silently omitted (no error output).
8. **Suggestion added.** When pending proposals exist, a suggestion line appears: "N soul change(s) staged for review".
9. **No side effects.** morning.ts remains read-only — no file mutations, no brain.db writes.

## Constraints

- morning.ts must remain fast. The soul_changes query and staged file scan should add minimal latency (< 100ms combined).
- Do NOT add interactive approval to the morning briefing. Review happens in Obsidian or via a separate command. The briefing is read-only.
- Use `getRecentSoulChanges()` from AAR-0003 (exported via brain/queries.ts -> brain/index.ts).
- Use `getPendingProposals()` from AAR-0005 (src/libs/staged.ts).
- Error handling: wrap both data sources in try/catch. A failure in one does not prevent the other from displaying.

## Out of Scope

- Interactive approval/rejection from the briefing
- Detailed gate/judge results in the briefing (too verbose — user reads the staged file in Obsidian for details)
- Historical change log (only shows last 24 hours of auto-applied changes)
- Any modifications to the `/morning` skill file — only the tool implementation changes
