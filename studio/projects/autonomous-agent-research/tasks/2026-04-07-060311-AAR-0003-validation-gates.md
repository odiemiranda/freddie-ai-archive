# AAR-0003: Validation Gates

## Objective

Create 5 validation gate functions that evaluate a proposed soul file change before it can be auto-applied. Each gate returns a pass/fail verdict with reasoning. All 5 must pass for the change to proceed to the judge panel.

Additionally, create the `soul_changes` table in brain.db to track drift over time (used by Gate 4).

## Context

These gates sit between the distill step (which proposes a soul diff) and the judge panel (which gives final approval). They are deterministic or semi-deterministic checks that catch structural problems before expensive LLM judge calls.

```
distill proposes diff → Gate 1 (Constitution) → Gate 2 (Regression) → Gate 3 (Size)
                       → Gate 4 (Drift) → Gate 5 (Safety) → [all pass?] → judge panel
```

If ANY gate fails, the change is staged for human review. No partial passes.

#### Types

```typescript
export interface GateResult {
  gate: string;           // gate name: "constitution" | "regression" | "size" | "drift" | "safety"
  pass: boolean;
  reason: string;         // human-readable explanation
}

export interface GateInput {
  agent: string;                    // which agent's soul file
  currentSoul: string;             // full text of current soul file
  proposedDiff: ProposedChange[];  // what distill wants to add/remove/modify
}

export interface ProposedChange {
  type: "add" | "remove" | "modify";
  section: string;          // which soul file section
  content: string;          // the text being added/removed/modified
  evidence: string;         // what knowledge supports this change
}

export interface AllGatesResult {
  passed: boolean;          // true only if ALL gates passed
  results: GateResult[];    // one per gate, in order
  failedGates: string[];   // names of gates that failed (empty if all passed)
}
```

#### Gate 1: Constitution

```typescript
/**
 * Gate 1: Constitution — LLM check against soul identity.
 * 
 * Sends the current soul file + proposed diff to Gemini.
 * Prompt asks: "Does this change contradict or dilute the agent's established character?"
 * 
 * Uses Gemini (cheapest, fastest) since this is a single yes/no assessment.
 */
export async function gateConstitution(input: GateInput): Promise<GateResult>
```

**Implementation:**
- Call `generateText()` from `src/libs/gemini.ts` with model `gemini-2.5-flash`, temperature 0.2
- System prompt: include the full current soul file
- User prompt: "The following change is proposed for {agent}'s soul file. Does this addition contradict, dilute, or conflict with the agent's established character, philosophy, or hard rules? Respond with ONLY one of: PASS (if the change is consistent) or FAIL: {reason} (if it contradicts the identity)."
- Parse response: if starts with "PASS" -> pass. If starts with "FAIL" -> fail with the reason after "FAIL:".
- Caller: `"gate-constitution"`

#### Gate 2: Regression

```typescript
/**
 * Gate 2: Regression — Semantic contradiction check.
 * 
 * Extracts existing rules/instincts from the soul file, embeds them,
 * embeds the proposed addition, and checks for contradictions.
 * A contradiction is defined as: >0.85 cosine similarity but semantically
 * opposite meaning (e.g., "always verify" vs "trust without checking").
 * 
 * Uses local ONNX embeddings (no API cost).
 */
export async function gateRegression(input: GateInput): Promise<GateResult>
```

**Implementation:**
1. Extract rule-like sentences from the current soul file:
   - Lines that start with `-`, `*`, or a number followed by a period
   - Lines that contain strong verbs: "always", "never", "must", "don't"
   - Sentences in sections named "Hard Rules", "Core Opinions", "Postmortem Reflexes"
   - Function: `extractRules(soulText: string): string[]`
2. For each proposed change content:
   - Embed the proposed text using `embedText()` from `src/libs/embeddings.ts`
   - Embed each extracted rule using `embedBatch()`
   - Compute cosine similarity between proposed text and each rule
   - If any pair has similarity > 0.85:
     - Make an LLM call to Gemini to confirm semantic contradiction: "Rule A says: '{existing}'. Proposed addition says: '{new}'. Are these contradictory? Respond CONTRADICTION or COMPATIBLE."
     - If Gemini says CONTRADICTION -> fail with the specific conflicting rule
3. The 0.85 threshold catches near-duplicates and potential contradictions. The LLM confirmation step prevents false positives from embeddings (two rules can be highly similar without contradicting).
- Helper: `cosineSimilarity(a: number[], b: number[]): number` — copy from existing `src/libs/brain/queries.ts` or import it.

#### Gate 3: Size

```typescript
/**
 * Gate 3: Size — Change magnitude check.
 * 
 * Purely deterministic. No LLM calls.
 * Max 3 lines added per cycle, max 1 removed, no modifications to existing lines.
 */
export async function gateSize(input: GateInput): Promise<GateResult>
```

**Implementation:**
- Count lines across all proposed changes:
  - `totalAdded`: sum of `content.split('\n').length` for all changes with `type === "add"`
  - `totalRemoved`: count of changes with `type === "remove"` (each removal = 1 logical change)
  - `totalModified`: count of changes with `type === "modify"`
- Rules:
  - `totalAdded > 3` -> fail: "Too many lines added ({n}). Max 3 per cycle."
  - `totalRemoved > 1` -> fail: "Too many removals ({n}). Max 1 per cycle."
  - `totalModified > 0` -> fail: "Modifications to existing lines require human review."
- If all pass -> pass: "Change size within limits: {added} added, {removed} removed."

#### Gate 4: Drift

```typescript
/**
 * Gate 4: Drift — Rolling window change rate check.
 * 
 * Queries soul_changes table for cumulative changes in the last 30 days.
 * If total changes exceed 10% of the soul file's line count, blocks.
 * 
 * Purely deterministic (database query + math). No LLM calls.
 */
export async function gateDrift(input: GateInput): Promise<GateResult>
```

**Implementation:**
1. Count total lines in current soul file: `const totalLines = input.currentSoul.split('\n').length`
2. Query `soul_changes` table for the agent's changes in the last 30 days:
   ```sql
   SELECT SUM(lines_added) as added, SUM(lines_removed) as removed
   FROM soul_changes
   WHERE agent = ? AND date >= date('now', '-30 days')
   ```
3. Calculate: `const cumulativeChanges = (added ?? 0) + (removed ?? 0)`
4. Calculate proposed: `const proposedAdded = /* count from input.proposedDiff */`
5. `const totalDrift = cumulativeChanges + proposedAdded`
6. `const driftPercent = (totalDrift / totalLines) * 100`
7. If `driftPercent > 10` -> fail: "Drift limit reached: {driftPercent.toFixed(1)}% changes in 30 days (max 10%). {cumulativeChanges} existing + {proposedAdded} proposed."
8. If pass -> pass: "Drift within limits: {driftPercent.toFixed(1)}% of soul file changed in 30 days."

**Note:** The `soul_changes` table is created in this task (see brain.db schema section below). It starts empty, so Gate 4 will pass until changes accumulate.

#### Gate 5: Safety

```typescript
/**
 * Gate 5: Safety — Protected section check.
 * 
 * Parses the soul file for protected sections and blocks changes to them.
 * Protected sections are identified by:
 * - Section names: "Who I Am", "Who You Are", "Creative Philosophy", "Hard Rules"
 * - HTML marker: <!-- protected -->
 * 
 * Purely deterministic. No LLM calls.
 */
export async function gateSafety(input: GateInput): Promise<GateResult>
```

**Implementation:**
1. Parse section headers from the soul file (`## ` headings)
2. Build a set of protected section names (case-insensitive match):
   ```typescript
   const PROTECTED_NAMES = [
     "who i am", "who you are", "creative philosophy", "hard rules",
     "the line i won't cross", "my line", "core identity"
   ];
   ```
3. Also scan for `<!-- protected -->` markers — any section containing this marker is protected
4. For each proposed change:
   - Normalize the change's `section` field to lowercase
   - Check if it matches any protected section name (substring match, not exact)
   - If match -> fail: "Cannot auto-modify protected section: '{section}'"
5. If no protected sections targeted -> pass: "No protected sections affected."

#### Orchestrator

```typescript
/**
 * Run all 5 gates sequentially on a proposed soul change.
 * Short-circuits: stops at first failure (no point running expensive checks
 * after a deterministic gate already failed).
 * 
 * Gate order is deliberate:
 * 1. Safety (fast, deterministic) — protects immutable sections
 * 2. Size (fast, deterministic) — catches oversized changes
 * 3. Drift (fast, DB query) — catches cumulative drift
 * 4. Regression (medium, embeddings + optional LLM) — catches contradictions
 * 5. Constitution (slow, LLM call) — deep identity check
 * 
 * Cheapest/fastest gates run first to avoid unnecessary LLM calls.
 */
export async function runAllGates(input: GateInput): Promise<AllGatesResult>
```

**Implementation:**
- Run gates in this order: safety, size, drift, regression, constitution
- On first failure: still run remaining gates (do NOT short-circuit — collect all failures for better human review context)
- Actually, reconsider: run all 5 always. The cost of gates 4 and 5 is one embedding batch + one Gemini call. That's cheap. And having all gate results helps the human reviewer understand the full picture when reviewing staged changes.
- Return `{ passed: results.every(r => r.pass), results, failedGates: results.filter(r => !r.pass).map(r => r.gate) }`

### 5. Helper: `logSoulChange()` in `src/libs/brain/queries.ts`

Add a function to log soul changes:

```typescript
export interface SoulChangeInput {
  agent: string;
  date: string;                  // YYYY-MM-DD
  linesAdded: number;
  linesRemoved: number;
  changeSummary: string;
  gateResults?: string;          // JSON string
  judgeResults?: string;         // JSON string
  status: "auto-applied" | "staged" | "rejected";
}

/** Log a soul file change for drift tracking. */
export function logSoulChange(input: SoulChangeInput): number {
  const raw = getRawDb();
  const ts = new Date().toISOString();

  const result = raw.prepare(`
    INSERT INTO soul_changes (agent, date, lines_added, lines_removed, change_summary, gate_results, judge_results, status, created_at)
    VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?)
  `).run(
    input.agent, input.date, input.linesAdded, input.linesRemoved,
    input.changeSummary, input.gateResults ?? null, input.judgeResults ?? null,
    input.status, ts
  );

  return Number(result.lastInsertRowid);
}

/** Query cumulative drift for an agent over a rolling window. */
export function querySoulDrift(agent: string, windowDays = 30): { added: number; removed: number } {
  const raw = getRawDb();
  const row = raw.prepare(`
    SELECT COALESCE(SUM(lines_added), 0) as added,
           COALESCE(SUM(lines_removed), 0) as removed
    FROM soul_changes
    WHERE agent = ? AND date >= date('now', '-' || ? || ' days')
  `).get(agent, windowDays) as { added: number; removed: number };

  return row;
}

/** Get recent soul changes for an agent (for morning briefing). */
export function getRecentSoulChanges(
  agent?: string,
  opts?: { limit?: number; since?: string }
): Array<{
  id: number; agent: string; date: string;
  linesAdded: number; linesRemoved: number;
  changeSummary: string; status: string; createdAt: string;
}> {
  const raw = getRawDb();
  const limit = opts?.limit ?? 20;
  let sql = "SELECT * FROM soul_changes";
  const params: any[] = [];
  const conditions: string[] = [];

  if (agent) {
    conditions.push("agent = ?");
    params.push(agent);
  }
  if (opts?.since) {
    conditions.push("date >= ?");
    params.push(opts.since);
  }

  if (conditions.length > 0) sql += " WHERE " + conditions.join(" AND ");
  sql += " ORDER BY created_at DESC LIMIT ?";
  params.push(limit);

  return raw.prepare(sql).all(...params) as any[];
}
```

## Constraints

- `gates.ts` does NOT call `distill-soul.ts` — it receives proposed changes as input.
- `gates.ts` does NOT write to the `staged/` directory — that is AAR-0005.
- `gates.ts` does NOT call the judge panel — that is AAR-0004.
- The `soul_changes` table is write-only from the gates' perspective. Gates read via `querySoulDrift()`, but the actual logging happens in the orchestration layer (which wires gates + judge + apply together — a post-task integration step).
- Use `import { generateText } from "./gemini.js"` for Gate 1, not the tool CLI.
- Use `import { embedText, embedBatch } from "./embeddings.js"` for Gate 2, not Gemini API embeddings.
