# AAR-0004: LLM Judge Panel

## Objective

Create a 3-model judge panel that evaluates proposed soul file changes after all validation gates pass. The panel calls Gemini, MiniMax, and Grok in parallel. Unanimous approval is required. One rejection stages the change for human review.

## Context

The judge panel is the final quality filter before a soul file change is auto-applied. It serves a different purpose than the validation gates:

- **Gates** are structural/mechanical checks (size, drift, contradictions, protected sections).
- **Judges** assess semantic quality: Is this change well-evidenced? Is it genuinely useful? Does it belong in the soul?

Using 3 different model providers (Gemini, MiniMax, Grok) provides diversity of perspective. Agreement across providers is a stronger signal than agreement within a single provider's model family.

```
gates pass → judge-panel.ts → Gemini + MiniMax + Grok (parallel)
                             → all APPROVE? → auto-apply
                             → any REJECT? → stage for human review
```

```typescript
export type JudgeVote = "approve" | "reject";

export interface JudgeVerdict {
  model: string;           // "gemini" | "minimax" | "grok"
  vote: JudgeVote;
  reason: string;          // explanation for the vote
  latencyMs: number;       // how long this judge took
}

export interface JudgePanelResult {
  approved: boolean;       // true only if ALL judges voted approve
  votes: JudgeVerdict[];   // one per model
  unanimous: boolean;      // same as approved (all must agree)
  dissents: string[];      // model names that voted reject
}

export interface JudgePanelInput {
  agent: string;
  currentSoul: string;              // full text of current soul file
  proposedDiff: string;             // human-readable diff description
  evidence: string;                 // knowledge file excerpts supporting the change
  gateResults: import("./gates.js").AllGatesResult;  // gate verdicts for context
}
```

#### Judge Prompt

All three judges receive the same prompt (fairness — no model gets extra context):

```typescript
function buildJudgePrompt(input: JudgePanelInput): string {
  return `You are a judge reviewing a proposed change to an AI agent's soul file (core identity document).

## Agent: ${input.agent}

## Current Soul File
${input.currentSoul}

## Proposed Change
${input.proposedDiff}

## Evidence (from knowledge files)
${input.evidence}

## Validation Gate Results
All structural gates passed:
${input.gateResults.results.map(r => `- ${r.gate}: ${r.pass ? "PASS" : "FAIL"} — ${r.reason}`).join("\n")}

## Your Task

Evaluate whether this soul file change should be automatically applied. Consider:

1. **Evidence quality:** Is the change supported by sufficient evidence from multiple sessions/workflows? A single observation is not enough for a soul change.
2. **Non-contradiction:** Does this change coexist with existing soul traits? (The regression gate checked embedding similarity, but you should check semantic compatibility at a deeper level.)
3. **Utility:** Will this change genuinely improve the agent's future work? Or is it trivial, situational, or already implied by existing traits?
4. **Proportionality:** Is the magnitude of the change proportional to the evidence? Small, proven refinements are good. Large shifts need overwhelming evidence.

## Response Format

Respond with EXACTLY one line in this format:
APPROVE: {brief reason why this change is safe and beneficial}
or
REJECT: {brief reason why this change should not be auto-applied}

Do not include any other text.`;
}
```

#### Individual Judge Calls

```typescript
import { generateText as geminiGenerate } from "./gemini.js";
import { chatCompletion as minimaxChat } from "./minimax.js";
import { chatCompletion as grokChat } from "./grok.js";

async function callGeminiJudge(prompt: string): Promise<JudgeVerdict> {
  const start = Date.now();
  const result = await geminiGenerate(prompt, {
    model: "gemini-2.5-flash",
    temperature: 0.2,         // low temp for consistent judgment
    maxOutputTokens: 256,     // short response — just APPROVE/REJECT + reason
    caller: "judge-panel-gemini",
  });
  return parseVerdict("gemini", result.text, Date.now() - start);
}

async function callMinimaxJudge(prompt: string): Promise<JudgeVerdict> {
  const start = Date.now();
  const result = await minimaxChat({
    model: "MiniMax-M2.5",    // cheaper model — judgment doesn't need M2.7
    messages: [{ role: "user", content: prompt }],
    temperature: 0.2,
    maxTokens: 256,
    caller: "judge-panel-minimax",
  });
  return parseVerdict("minimax", result.text, Date.now() - start);
}

async function callGrokJudge(prompt: string): Promise<JudgeVerdict> {
  const start = Date.now();
  const result = await grokChat({
    model: "grok-4-1-fast-non-reasoning",
    messages: [{ role: "user", content: prompt }],
    temperature: 0.2,
    maxTokens: 256,
    caller: "judge-panel-grok",
  });
  return parseVerdict("grok", result.text, Date.now() - start);
}
```

#### Response Parser

```typescript
function parseVerdict(model: string, text: string, latencyMs: number): JudgeVerdict {
  const trimmed = text.trim();

  // Try to parse structured response
  if (trimmed.toUpperCase().startsWith("APPROVE")) {
    const reason = trimmed.replace(/^APPROVE:?\s*/i, "").trim() || "Approved without specific reason";
    return { model, vote: "approve", reason, latencyMs };
  }

  if (trimmed.toUpperCase().startsWith("REJECT")) {
    const reason = trimmed.replace(/^REJECT:?\s*/i, "").trim() || "Rejected without specific reason";
    return { model, vote: "reject", reason, latencyMs };
  }

  // Fallback: if response doesn't follow format, treat as reject (conservative)
  // This prevents a confused model from accidentally approving
  return {
    model,
    vote: "reject",
    reason: `Could not parse judge response. Raw: "${trimmed.slice(0, 200)}"`,
    latencyMs,
  };
}
```

#### Main Panel Function

```typescript
/**
 * Run the 3-model judge panel.
 * All three models are called in parallel via Promise.allSettled.
 * 
 * If a model fails (API error, timeout), it counts as a rejection.
 * This is the conservative choice — infrastructure failures should not
 * enable auto-application of soul changes.
 * 
 * @param input - The soul change to evaluate
 * @returns Panel result with all votes
 */
export async function runJudgePanel(input: JudgePanelInput): Promise<JudgePanelResult> {
  const prompt = buildJudgePrompt(input);

  // Call all three judges in parallel
  const results = await Promise.allSettled([
    callGeminiJudge(prompt),
    callMinimaxJudge(prompt),
    callGrokJudge(prompt),
  ]);

  const votes: JudgeVerdict[] = results.map((result, i) => {
    const modelNames = ["gemini", "minimax", "grok"];
    if (result.status === "fulfilled") {
      return result.value;
    }
    // API failure = reject (conservative)
    return {
      model: modelNames[i],
      vote: "reject" as JudgeVote,
      reason: `API error: ${result.reason?.message ?? "unknown error"}`,
      latencyMs: 0,
    };
  });

  const dissents = votes.filter(v => v.vote === "reject").map(v => v.model);
  const approved = dissents.length === 0;

  return {
    approved,
    votes,
    unanimous: approved,
    dissents,
  };
}
```

## Acceptance Criteria

1. **3 models called in parallel.** `Promise.allSettled` is used, not sequential calls.
2. **Unanimous approval required.** `approved === true` only when all 3 votes are "approve".
3. **API failures count as reject.** A timeout, 429, or other API error results in a "reject" verdict for that model.
4. **Parse failures count as reject.** If a model returns gibberish instead of "APPROVE: ..." or "REJECT: ...", it is treated as a rejection.
5. **Low temperature.** All judges use temperature 0.2 for consistency.
6. **Short responses.** maxOutputTokens/maxTokens is 256 — judges provide a vote and brief reason, not an essay.
7. **Latency tracked.** Each verdict includes `latencyMs` so the orchestrator can log performance.
8. **Same prompt for all models.** `buildJudgePrompt()` is called once, result passed to all three judge functions.
9. **Caller tracking.** Each judge call uses a distinct caller name for usage tracking: `judge-panel-gemini`, `judge-panel-minimax`, `judge-panel-grok`.

## Constraints

- This module does NOT decide what to do with the result (apply vs stage). That is the orchestration layer.
- This module does NOT write to brain.db or the filesystem. It returns the result for the caller to handle.
- This module does NOT read soul files from disk. It receives the soul file content as input.
- Use the existing library functions (`generateText` from gemini.ts, `chatCompletion` from minimax.ts and grok.ts). Do NOT create new API client code.
- Use `MiniMax-M2.5` (not M2.7) for cost efficiency. The judgment task is simple enough.
- Use `grok-4-1-fast-non-reasoning` for Grok. No need for the reasoning model on a yes/no judgment.

## Design Decisions

**Why unanimous?** A single dissent means at least one model's training data / reasoning produces a different conclusion. For identity changes, the cost of a false positive (bad change auto-applied) far exceeds the cost of a false negative (good change delayed for human review). Unanimous is the correct bar.

**Why treat API failures as rejections?** Same principle. If we can't get a vote, we default to the safer option. The human will see the staged proposal next morning and can approve manually.

**Why low temperature?** Judges should be consistent. High temperature introduces randomness in the vote, which means the same change might pass on retry. That undermines the gate's purpose.

**Why `Promise.allSettled` instead of `Promise.all`?** `Promise.all` short-circuits on first rejection. We want all three votes even if one fails — the human reviewer benefits from seeing all perspectives.

## Out of Scope

- Retry logic for failed API calls (a failure is a rejection — simple, safe)
- Weighted voting (all models have equal weight)
- Configuring which models serve as judges
- Caching judge responses
- Any UI for reviewing judge decisions (that's the morning briefing, AAR-0006)
