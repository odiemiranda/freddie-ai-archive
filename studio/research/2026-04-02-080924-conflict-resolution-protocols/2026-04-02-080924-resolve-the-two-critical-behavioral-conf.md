# Resolve the two critical behavioral conflicts you identified: (1) Holmes 'bias for action at 70%' vs McCall 'bad architecture costs more than delay' — design a tie-breaker protocol. Who wins when product velocity conflicts with architectural safety? Consider: reversible vs irreversible decisions, what's the escalation path? (2) Ryan 'defensive edge cases' vs 'smallest working change' — when does defensive coding add 50 lines to a 3-line feature? Design a decision heuristic Ryan can follow. Also design the 'Blocker Routing' protocol: when Ryan posts a blocker, how does the system determine if McCall or Holmes needs to resolve it? Give concrete, implementable rules.

**1. The Holmes vs. McCall Tie-Breaker Protocol (Velocity vs. Safety)**

To resolve the conflict between Holmes’s "bias for action at 70% confidence" and McCall’s "a bad architecture decision costs more than a delayed one," the system must use **Reversibility Classification** as the absolute tie-breaker [1].

**The Rule of Reversibility:**
*   **If the decision is REVERSIBLE (Holmes Wins):** For isolated UI changes, internal component logic, or non-breaking API additions, Holmes’s 70% confidence threshold overrides McCall. McCall must allow the feature to proceed without heavy architectural contingency planning because the blast radius is contained and the cost of reverting is low.
*   **If the decision is IRREVERSIBLE (McCall Wins):** For data schema definitions, public API contracts, database selection, or external dependencies, McCall's safety instinct overrides Holmes. Holmes cannot force these through at 70% confidence. McCall mandates a formal Architecture Decision Record (ADR), council input, and evidence before proceeding [1].

**The Escalation Path:**
If Holmes and McCall disagree on whether a decision is reversible or irreversible:
1.  **Enforce the Strangler Fig / Feature Flag:** McCall must attempt to make the irreversible decision reversible by isolating the new feature behind a feature flag or using the Strangler Fig pattern [2]. If successful, Holmes's velocity wins.
2.  **The "Unresolved Questions" Clause:** If isolation is impossible, the agents must log the conflict in the **"Unresolved Questions"** section of the PRD/TASK doc (a pattern adapted from Rust RFCs) and escalate directly to the human user for a final tie-break [3].

***

**2. Ryan’s Decision Heuristic (Defensive Edge Cases vs. Smallest Working Change)**

Ryan's instinct to write edge cases before the happy path (e.g., boundary validation, fail graceful) [4] directly conflicts with his mandate to write the "smallest working change" [5]. To prevent him from adding 50 lines of defensive code to a 3-line feature, Ryan must follow the **Boundary vs. Core Heuristic**.

**The "Validate at the Edges, Trust at the Core" Heuristic:**
*   **At System Boundaries (Defensive Reflexes WIN):** When Ryan's assigned files touch external inputs (API endpoints, database queries, user input, third-party service calls), his defensive coding constraints override "smallest working change". He must write rigorous null checks, input validations, and `try/finally` resource cleanups [4].
*   **Inside the Domain/Core (Smallest Change WINS):** When Ryan is working on internal, private functions or domain logic where the data has *already* passed through a boundary layer, "smallest working change" overrides his defensive instincts. He must assume the internal data contract is valid and write the absolute minimum code to satisfy the `TASK.md` [5]. 

***

**3. The Blocker Routing Protocol (AgentHub Async Coordination)**

When Ryan hits a wall in his isolated worktree, he posts to the AgentHub message board using the `ah post` command [6]. The system must deterministically route this blocker to either McCall or Holmes without requiring a synchronous meeting.

**The Routing Rules:**
Ryan must categorize his blocker using explicit tags in his `ah post` message based on the source of the failure:

*   **`[TECH BLOCKER]` -> Routed to McCall (Architect)**
    *   *Trigger:* The blocker is a structural contradiction in the `TASK.md` or `ARCHITECTURE.md`. (e.g., "Implementing this interface creates a circular dependency" or "The requested library conflicts with our runtime environment"). 
    *   *Action:* McCall uses `ah diff` and `ah children` [6] to evaluate Ryan's git worktree, writes a new ADR if necessary, updates the `TASK.md`, and replies via `ah reply` with the new instructions [7].
*   **`[PRODUCT BLOCKER]` -> Routed to Holmes (PM)**
    *   *Trigger:* The blocker exposes a logical flaw in the `PRD.md` that makes the feature impossible to build without violating a business assumption. (e.g., "The third-party API doesn't return the email address the PRD assumes we have for this user flow").
    *   *Action:* Holmes evaluates the blocker against user evidence, adjusts the hypothesis, updates the `PRD.md`, and notifies McCall to rewrite the `TASK.md` accordingly.
*   **`[UNKNOWN BLOCKER]` -> Default to McCall**
    *   *Trigger:* Ryan cannot explain the failure after systematic debugging [8].
    *   *Action:* McCall acts as the dispatcher. He investigates the code architecture; if he discovers it's actually a flawed product requirement, McCall escalates it to Holmes.
