# Summarize the key findings for daemon architecture: chokidar Windows reliability, Bun worker_threads maturity, SQLite task queue lease semantics, Windows daemon lifecycle without service, and file watcher debounce. What are the top architectural recommendations?

**Chokidar Windows Reliability**
Native `fs.watch` on Windows relies on the `ReadDirectoryChangesW` API, which is notoriously unreliable and drops events when rapid bulk changes occur [1]. When sync clients or editors use atomic saves (writing to a temp file and renaming), they trigger a noisy burst of file system events [2, 3]. Furthermore, upgrading to **Chokidar v4.0.0 on Windows causes critical regressions**, such as failing to detect new files entirely and misreporting atomic writes as `unlink` events [4-6]. The only guaranteed way to avoid missed events on Windows network drives or complex setups is to enable **polling (`usePolling: true`)**, though this severely impacts CPU and memory consumption [7-9].

**Bun `worker_threads` Maturity**
As of 2025/2026, Bun's `worker_threads` implementation remains **experimental and highly unstable** [10]. It suffers from severe memory leaks, extreme virtual memory commits (e.g., 47GB on Windows), and frequent segmentation faults during worker termination or high load [11-14]. Additionally, Bun is missing support for the `stdout` and `stderr` options in the `Worker` constructor, which actively breaks compatibility with major ecosystem libraries like `workerpool` and `BullMQ` [15, 16]. 

**SQLite Task Queue Lease Semantics**
A robust SQLite task queue requires **Write-Ahead Logging (WAL)** to allow concurrent readers and writers, alongside opening exactly one connection per worker [17-19]. To prevent multiple workers from checking out the same job, the system must use a lease (or lock). **A short time-bounded lease (e.g., 30 seconds) combined with continuous heartbeat renewals is the optimal strategy** [20-22]. If a task is highly CPU-intensive or a worker crashes prematurely, the worker fails to send the heartbeat, the lock naturally expires, and the queue marks the job as "stalled" so it can be safely reattempted by another worker [21, 23, 24]. To avoid `SQLITE_BUSY` deadlocks during the lock acquisition itself, the worker must use `BEGIN IMMEDIATE` to instantly promote the transaction to a write lock [25-27].

**Windows Daemon Lifecycle Without a Service**
For a single-user developer scenario, avoid system-wide wrappers like NSSM that require administrative privileges, and instead use **Windows Task Scheduler** or a user-scoped process manager like **PM2** [28-30]. However, Task Scheduler has a massive hidden performance penalty: it defaults background tasks to a low I/O priority and a memory priority of 2, making processes run significantly slower than interactive sessions [31, 32]. You must export the task as an XML file, **change the `<Priority>7</Priority>` tag to `4`**, and re-import it to restore normal CPU, I/O, and memory priorities [32, 33]. For modern lifecycle management with live logging, graceful shutdowns, and process-tree zombie prevention, newer tools like **Servy** vastly outperform legacy wrappers [34-36].

**File Watcher Debounce Strategies**
Because file systems emit multiple raw events for a single save (flushing content, updating timestamps, atomic renames), a simple time-based debounce (which only emits the last event) will drop critical context [3]. Production watchers must use **event coalescing**, which batches events over a short window, drops `Modify` events that immediately follow `Create` events, and logically stitches together `Rename` events into a single actionable update [37]. The window duration must match the workflow: **100-200ms for fast IDE reloads, 500-1000ms for build triggers, and 2-5 seconds for slower backup operations** [38]. 

**Top Architectural Recommendations**
1.  **Use Child Processes over Worker Threads in Bun:** Until Bun resolves its core segmentation faults and missing `stdout` APIs, spawn standard child processes and use IPC for parallel work on Windows [39].
2.  **Downgrade to Chokidar v3.6.0 on Windows:** To avoid the missed file detection bugs in v4.x, stick to v3.6.0 or implement a coalescing debouncer (like Rust's `notify_debouncer_full`) to safely handle atomic save artifacts [4, 37, 40].
3.  **Implement Heartbeat-Driven Leases for SQLite Queues:** Never use long global timeouts (e.g., 4 hours) for tasks. Use a 30-to-60-second lock duration and have the worker periodically update a `last_active` timestamp (heartbeat) to ensure rapid crash recovery without duplicate processing [20, 22, 41]. 
4.  **Edit Task Scheduler XML for Daemons:** If running background daemons via Task Scheduler, always modify the XML to `<Priority>4</Priority>` to prevent the OS from aggressively throttling your script's memory and I/O [32].
