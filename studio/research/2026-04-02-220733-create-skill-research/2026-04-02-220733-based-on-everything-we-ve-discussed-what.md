# Based on everything we've discussed: what supporting tools does our create-skill need beyond the main scaffolder? For example: a skill validator, a skill linter, a skill tester? Or is validation just part of the main tool? What's the minimum toolset that provides real value without over-engineering? Remember: we're a single-user CLI-first system, not a marketplace or team distribution platform.

For your single-user, CLI-first system, validation should **not** be tightly coupled into the main scaffolder. Decoupling validation into standalone CLI scripts allows you to run these checks on your 20+ existing skills independently, rather than only when creating new ones [1]. 

To provide maximum value without over-engineering, your minimum supporting toolset should consist of three focused utilities:

**1. A Spec Validator & Linter**
You need a standalone linter similar to the `agnix` tool or `validate.py` [1, 2]. This tool verifies that your `SKILL.md` files comply with the Agent Skills open standard by checking frontmatter formats, naming constraints (lowercase, hyphens, under 64 characters), and ensuring descriptions aren't truncated [1, 3]. Crucially, it should include logic similar to `claude-rules-doctor` to scan for "dead rules"—instances where `paths:` globs in your frontmatter no longer match any actual files due to renaming, which causes skills to silently fail [4].

**2. A Staleness & Drift Checker**
When managing a large ecosystem of multi-agent workflows alone, skills break silently over time. A tool modeled after `staleness_check.py` is essential for catching this rot before execution [1]. It should perform three checks:
*   **Review tracking:** Reads the `last_reviewed` and `review_interval_days` frontmatter to flag skills that haven't been audited recently [5, 6].
*   **Dependency health:** Uses a `--check-deps` flag to ping any external URLs or documentation links your skills rely on to ensure they are still reachable [7].
*   **Schema drift:** Uses a `--check-drift` flag to verify that external APIs still return the expected JSON keys, preventing your sub-agents from failing due to unannounced API changes [6, 7].

**3. A Local Sync Script**
Because you are a solo user, you do not need a team registry, but you do need to maintain visibility of your tools. A simple script like `sync-skills.js` is the perfect lightweight solution [8]. It scans your local skills directory and automatically updates your main `README.md` with an accurate table of available skills [8, 9]. You can adapt this to additionally sync your skill metadata directly into your `brain.db` so Freddie and McCall always have an up-to-date index of the system's capabilities [10].

**What to explicitly AVOID (Over-engineering)**
*   **Team Registries & Cross-Platform Installers:** Do not build shared git registries (`skill_registry.py`) or cross-platform formatting scripts (`install.sh`) designed to port skills to Cursor, Windsurf, or Copilot [11-13]. These add immense complexity and provide zero value to a strictly Claude Code CLI-first environment.
*   **Heavy Benchmarking Pipelines:** Tools like Skill Forge include complex `eval` and `benchmark` scripts for variance analysis and grading [14, 15]. You should skip these. Your existing `workflow.md` architecture already implements a phase-gated "tri-critic" system (Gemini, MiniMax, Grok) that evaluates and reconciles output quality dynamically during the execution phase [10, 16]. Building a separate offline benchmarking tool would duplicate the robust quality-control logic already running inside your orchestrator.
