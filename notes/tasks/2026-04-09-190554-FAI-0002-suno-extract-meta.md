# FAI-0002 — suno-extract-meta

## Context

Session 2026-04-09: user discovered Suno added an "upload audio" feature that analyzes a track and auto-populates Lyrics + Style in the generation UI. The extracted values round-trip well — regenerating from them produced a close match to the original audio. This is the first time we have access to Suno's own semantic interpretation of a track (as opposed to our measured data from librosa plus hand-written Tala craft descriptors).

The 7-step flow was reverse-engineered from a real HAR file on the user's machine (`C:/Users/odiem/Downloads/suno.com.har`, NOT committed, contains live session auth). Two explorer passes mapped all request/response shapes.

This task builds a standalone v1 CLI tool. Music card pipeline integration is a follow-up task — **do not** include it here.

## What to Build

A single CLI tool — `src/tools/suno-extract-meta.ts` — that takes a local mp3 path, runs the full 7-step Suno upload-and-analyze flow, and returns Suno's extracted Lyrics (typed-bracket format) + Style (rich prose) for the track. Invoked via `bun run tool suno-extract-meta <path.mp3>`.

### High-level data flow

```
local.mp3
   │
   ▼
[1] POST /api/uploads/audio/           → {upload_id, s3_url, s3_fields}
   │
   ▼
[2] POST s3_url (multipart, no Suno auth, 3x retry w/ backoff) → 204
   │
   ▼
[3] POST /api/uploads/audio/{upload_id}/upload-finish/   → ack
   │
   ▼
[4] GET  /api/uploads/audio/{upload_id}/ (poll 1s, 90s timeout)
         processing → passed_audio_processing → passed_artist_moderation → complete
   │
   ▼
[5] POST /api/uploads/audio/{upload_id}/initialize-clip/   → {clip_id, image_url, ...}
   │                                                           ^^^ NEW id, not upload_id
   ▼
[6] POST /api/gen/{clip_id}/set_metadata/         ← TRAILING SLASH
         body: {title, image_url, is_audio_upload_tos_accepted: true}
         → {prompt: "<lyrics with typed brackets>"}
   │
   ▼
[7] POST /api/gen/{clip_id}/set_audio_description  ← NO TRAILING SLASH
         body: {gemini_description_accepted: true}
         → {metadata: {tags: "<prose style>", ...}, display_tags, audio_url, ...}
   │
   ▼
SunoExtractResult (pretty or --json)
```

### The 7 steps — request/response contracts

All Suno API calls use base `https://studio-api-prod.suno.com` and MUST include these headers:

| Header | Value |
|---|---|
| `browser-token` | `$SUNO_BROWSER_TOKEN` (env) |
| `device-id` | `$SUNO_DEVICE_ID` (env) |
| `origin` | `https://suno.com` |
| `referer` | `https://suno.com/` |
| `accept` | `application/json` |
| `content-type` | `application/json` (on JSON POSTs) |

The S3 upload (step 2) uses a DIFFERENT host entirely and must NOT carry Suno auth headers.

**Step 1 — Create upload session**
```
POST /api/uploads/audio/
body: {"extension": "mp3"}
→ { id, url, fields: {Content-Type, key, AWSAccessKeyId, policy, signature, ...}, is_file_uploaded: false }
```

**Step 2 — S3 upload (pre-signed POST)**
```
POST <url from step 1>
Content-Type: multipart/form-data
fields: (all fields from step 1, in order) + file: <binary>
→ 204 No Content
```

Critical: append `file` LAST in the FormData. Pass all step-1 fields verbatim — do not reorder, re-sign, or modify. Bun's `FormData` handles SigV4 pre-signed POSTs correctly when ordering is preserved.

**Step 3 — Notify upload complete**
```
POST /api/uploads/audio/{upload_id}/upload-finish/
body: {"upload_type": "file_upload", "upload_filename": "<original basename>.mp3"}
→ ack
```

**Step 4 — Poll for analysis**
```
GET /api/uploads/audio/{upload_id}/
→ { id, status, ... }
```
Poll every 1000ms. Timeout 90s. Proceed only on `status === "complete"`. Progression: `processing → passed_audio_processing → passed_artist_moderation → complete`. Any terminal non-complete status (`failed`, `moderation_rejected`, etc.) → fail loud, exit 6 with the last status included.

**Step 5 — Initialize clip**
```
POST /api/uploads/audio/{upload_id}/initialize-clip/
body: {}
→ { id: "<clip_id — NEW, NOT upload_id>", title, image_url, ... }
```

`clip_id` is what steps 6 and 7 use. Do not confuse with `upload_id`.

**Step 6 — Fetch lyrics (TRAILING SLASH MANDATORY)**
```
POST /api/gen/{clip_id}/set_metadata/
body: { title, image_url, is_audio_upload_tos_accepted: true }
→ { id, title, image_url, prompt: "<FULL LYRICS WITH TYPED BRACKETS>", video_is_stale, opt_out_video_cover_hook }
```
The `prompt` field is the target. Suno has already generated lyrics during step 4 polling (Gemini on their backend); this call acknowledges and returns them.

**Step 7 — Fetch style prose (NO TRAILING SLASH)**
```
POST /api/gen/{clip_id}/set_audio_description
body: { gemini_description_accepted: true }
→ {
    status: "complete",
    id, title, audio_url, image_url,
    metadata: { tags: "<FULL PROSE STYLE>", prompt: "<lyrics>", type: "upload", duration, has_vocal, ... },
    display_tags: "alternative rock, britpop, male falsetto vocals",
    ...
  }
```

Targets: `metadata.tags` (rich prose Style block) and `display_tags` (short comma-separated).

### The trailing-slash asymmetry

Step 6 is `/api/gen/{clip_id}/set_metadata/` — trailing slash. Step 7 is `/api/gen/{clip_id}/set_audio_description` — no trailing slash. This is a Django REST Framework quirk and it WILL break if the path is templated naively as `` `/api/gen/${clip_id}/${action}/` ``. Use **two distinct URL constants** and comment the asymmetry. No clever templating.

### Architecture decisions

**A1. Single file unless it grows.** Start with everything in `src/tools/suno-extract-meta.ts`. If at end of Phase 1 the file exceeds ~400 lines, extract a helper module at `src/libs/suno-api/` containing typed interfaces, the 7-step functions, and a shared header builder. Ryan's call based on actual code size. Do not pre-split.

**A2. Bun `fetch` only.** No `axios`, `undici`, `node-fetch`, or any new dependency. Bun's global `fetch` handles JSON POSTs and multipart pre-signed S3 POSTs fine. `package.json` must be untouched.

**A3. Two distinct URL constants for steps 6 and 7.**
```ts
const URL_SET_METADATA = (clipId: string) =>
  `${BASE}/api/gen/${clipId}/set_metadata/`;        // trailing slash — DRF requires it
const URL_SET_AUDIO_DESCRIPTION = (clipId: string) =>
  `${BASE}/api/gen/${clipId}/set_audio_description`; // NO trailing slash — DRF 404s with it
```
Comment both with why they differ.

**A4. Fail closed on unexpected shapes.** Any missing field at step 6 (`prompt`) or step 7 (`metadata.tags`) → exit 7 with a specific message. Never emit partial output. Never silently fall back to an empty string.

**A5. Retry scope is narrow.** S3 upload only (step 2): 3 retries with exponential backoff at 1s, 2s, 4s. Every other Suno API call fails fast. The step-4 polling loop is its own mechanism and is not "retry" — it's the expected flow.

**A6. Secret handling.** `SUNO_BROWSER_TOKEN` and `SUNO_DEVICE_ID` NEVER appear in any stdout, stderr, or error message. Not even truncated, not even at a hypothetical `--debug` level. Error messages reference them by name only ("SUNO_BROWSER_TOKEN is not set"), never by value.

**A7. PII handling.** Response bodies contain `user_id`, `display_name`, `avatar_image_url`, and other PII. Only the fields that map into `SunoExtractResult` leave the tool. Raw bodies are not written to stdout. If a `--debug` flag is ever added (not in v1), it goes to stderr only.

### Tool interface

| Invocation | Behavior |
|---|---|
| `bun run tool suno-extract-meta <path.mp3>` | Full 7-step flow. Pretty markdown output on stdout. Exit 0 on success. |
| `bun run tool suno-extract-meta <path.mp3> --json` | Full 7-step flow. `SunoExtractResult` as single JSON object on stdout. Exit 0 on success. |
| `bun run tool suno-extract-meta --check-token` | Cheap authenticated GET against Suno. 200 → exit 0 "Token valid". 401 → exit 2 with expired-token message. Other → exit 3. NO upload, NO file arg. |
| `bun run tool suno-extract-meta --setup-help` | Prints step-by-step copy-paste instructions for extracting `SUNO_BROWSER_TOKEN` and `SUNO_DEVICE_ID` from Chrome DevTools. Zero network calls. Exit 0. |
| `bun run tool suno-extract-meta --help` | Usage line. Exit 0. |

### SunoExtractResult (JSON output shape)

```ts
interface SunoExtractResult {
  file: { path: string; filename: string; size_bytes: number };
  upload_id: string;
  clip_id: string;
  audio_url: string;      // Suno-hosted copy of our upload
  image_url: string;
  lyrics: string;         // step 6 metadata.prompt (typed-bracket format)
  style_prose: string;    // step 7 metadata.tags (full prose)
  display_tags: string;   // step 7 top-level display_tags
  duration_seconds: number;
  has_vocal: boolean;
  extracted_at: string;   // ISO 8601
}
```

## Architectural Constraints

1. **Bun + TypeScript only.** No Python, Node, or external runtimes.
2. **Bun global `fetch` only.** No new HTTP libraries. Bun handles multipart and pre-signed S3 POSTs natively.
3. **No new package.json dependencies for v1.**
4. **`toolDef` export required.** Tool must appear in `bun run tool --list`. Tool name: `suno-extract-meta`. Match file name exactly.
5. **Fail closed on all branches.** Unexpected response shapes → explicit error + specific exit code. Never partial output. Never silent fallback.
6. **Two distinct URL constants for set_metadata and set_audio_description.** Comment the trailing-slash asymmetry inline.
7. **Never log secrets.** `SUNO_BROWSER_TOKEN` and `SUNO_DEVICE_ID` values never touch stdout or stderr under any code path. Reference them by env var name only.
8. **PII minimization.** Only the `SunoExtractResult` fields leave the tool. Full response bodies are not printed. No `--debug` flag in v1.
9. **Never commit the HAR file.** `C:/Users/odiem/Downloads/suno.com.har` stays where it is. Do not copy it into the repo, not even as a test fixture.
10. **S3 retries are narrow.** 3 retries, exponential backoff 1s/2s/4s, step 2 only. No other retries. Polling is not retry.
11. **Validate inputs before any network call.** File existence + `.mp3` extension + env vars checked in that order before step 1 fires. Fail fast on invalid input; do not waste a Suno API call.
12. **Tool is discoverable but self-contained.** No imports from `src/tools/musiccard/`, no imports from other tools, no imports from skills.

## Notes for Ryan

1. **The trailing-slash asymmetry is real, non-obvious, and load-bearing.** Two distinct URL constants. Comment both. Test-assert both. If you template `` `/api/gen/${clip_id}/${action}/` `` you will break step 7 and spend an hour wondering why you're getting 404s from a path that "looks right."

2. **Step 5's `id` is a NEW id.** `clip_id !== upload_id`. Every place that calls step 6 or 7 must use clip_id. Every place that calls step 3 or 4 must use upload_id. Don't let them converge into one variable. Name them `uploadId` and `clipId` explicitly and never shadow.

3. **Steps 6 and 7 are acknowledge-and-return, not compute-and-return.** Suno computed the lyrics and style during step 4 polling using Gemini on their backend. Your POST bodies acknowledge acceptance; the response payload is what Suno already has in hand. This means both calls are fast (~500-800ms in the HAR) and their shapes are stable. No polling between or after them.

4. **S3 upload uses step-1 fields EXACTLY.** Do not reorder. Do not re-sign. Do not modify casing. Do not add your own `Content-Type` to the multipart parts. Append `file` last. Bun's `FormData` is correct if you append in the right order.

5. **`device-id` is its own header, mandatory on every Suno API call.** First explorer pass missed this. It is NOT derived from `browser-token`. Both must be present.

6. **Don't over-engineer URL constants.** Two `const` strings or two tiny arrow helpers. That's it. No URL builder class. No action enum.

7. **HAR stays in `C:/Users/odiem/Downloads/`.** Do not copy it. Do not reference it in test fixtures. Hand-transcribe the shapes you need into test code and strip auth. This is a hard rule — the HAR contains live session auth.

8. **Windows + Bun multipart gotcha.** We've hit file-path encoding weirdness with multipart on Windows before. Test the S3 upload path on Windows explicitly. If you see `ENOENT` or garbled bytes, check how the file is being read (prefer `Bun.file(path)` and pass the `File`/`Blob` directly to FormData — that's the Bun-native path).

9. **Input validation order:** file exists → extension is mp3 → env vars present. All three before any network call. Cheapest-first, so a typo in a path doesn't hit Suno's servers.

10. **The `vault/studio/references/shared/music-cards/audio/1774630111-jacob-collier-improvises-with.mp3` file is in the repo** and is the manual integration test target. Do NOT commit any output files generated from it during development.

11. **If you hit an ambiguity in this doc — stop and ask McCall.** Do not guess. This is a reverse-engineered API; guessing will produce silent shape mismatches that look like Suno bugs but are actually contract drift in this tool.
